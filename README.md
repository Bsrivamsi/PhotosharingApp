# PhotoTribe - Full-Stack Photo Sharing App

PhotoTribe is a full-stack photo sharing platform built with Spring Boot and React. It combines a polished public gallery, social interactions, profile management, moderation workflows, and analytics in a deployment-ready project structure.

## Highlights

- JWT authentication and profile management
- Photo upload with thumbnails, AI-style descriptions, and category inference
- Responsive public gallery with share, like, follow, comment, and download actions
- Role-based admin and moderator dashboard with moderation queue
- Analytics charts, activity logs, pagination, search, and filtering
- Local development and deployment support for Vite, Render, and Netlify

## Feature Breakdown

### User-facing features

- Register and login with JWT authentication
- Browse a public gallery combining curated sample photos and uploaded photos
- Upload photos with title, description, category, and live preview
- AI-style autogenerated one-line descriptions for uploads, with user override support
- Automatic category suggestions based on title and description keywords
- View photos by category and browse category-specific collections
- Download original photos and open full-size image links
- Share photos via Facebook, X, LinkedIn, Email, WhatsApp, and QR code
- Like photos, follow uploaders, and comment directly from gallery cards
- Manage profile bio and profile photo from the profile page
- Submit contact and feedback forms from the About page

### Admin and moderator features

- Role-based dashboard access for `ROLE_ADMIN` and `ROLE_MODERATOR`
- Admin analytics overview with chart visualizations and real category distribution
- Paginated management tables for users, photos, moderation queue, categories, and activity logs
- Search and filter support across users, photos, pending photos, categories, and logs
- Approve or reject pending photos with review notes
- Auto-approval scheduler for pending photos after 30 seconds without moderator action
- Admin-only user creation, role updates, soft delete, suspension, and batch deletion
- Admin-only category CRUD and admin photo upload flow
- Activity log browsing and batch admin actions tracking

## How To Run

### 1. Start Backend

Option A (Windows quick start):

```bat
cd backend
run-backend.bat dev
```

`run-backend.bat` accepts profile argument: `dev`, `test`, or `prod`.

Profile examples with one command:

```bat
run-backend.bat dev
run-backend.bat test
run-backend.bat prod
```

Option B (standard Maven):

```bash
cd backend
mvn spring-boot:run
```

Backend starts at:

- `http://localhost:8080`

### 2. Start Frontend

Open a new terminal:

```bash
cd frontend
npm install
npm run dev
```

Frontend starts at:

- `http://localhost:5173`

## How Auth Works

- Register/Login endpoints are served by backend under `/api/auth/*`.
- Frontend calls relative `/api` routes.
- In local development, Vite proxy forwards `/api` to backend (`http://localhost:8080`).
- On successful login/register, JWT token is saved in browser local storage.
- Profile data is loaded from `/api/auth/me` after login.
- Profile photo uploads are served by backend storage under `/api/auth/profile/photo/*`.

- `GET /api/photos` - list photos (public)

- `POST /api/photos/{id}/like` - toggle like for a photo

- `POST /api/comments/photo/{photoId}` - add a comment to a photo
- `GET /api/comments/photo/{photoId}` - list comments for a photo
- `PUT /api/comments/{commentId}` - update a comment
- `DELETE /api/comments/{commentId}` - delete a comment
- `PUT /api/comments/{commentId}/approve` - approve a comment
- `PUT /api/comments/{commentId}/reject` - reject a comment
- `POST /api/users/{username}/follow` - follow/unfollow a user
- `GET /api/users/{username}/followers` - get followers/following counts
- `GET /api/users/{username}/followers/check` - get current user's follow status
- `GET /api/categories` - list public categories
- `POST /api/contact` - submit a contact message
- `POST /api/feedback` - submit feedback and rating
- `GET /api/admin/analytics/dashboard` - admin analytics summary and chart data
- `GET /api/admin/analytics/history` - historical analytics series
- `GET /api/admin/analytics/export/csv` - export analytics CSV
- `GET /api/admin/users` - paginated users list with search/filter
- `POST /api/admin/users` - create a user
- `PUT /api/admin/users/{id}` - update user data or role
- `DELETE /api/admin/users/{id}` - soft delete user
- `POST /api/admin/users/{id}/suspend` - suspend user for 30 days
- `POST /api/admin/users/batch-delete` - batch soft delete users
- `GET /api/admin/photos` - paginated photos list with search/filter

## Local Development Notes

- Frontend dev server runs on `5173` and proxies `/api` to `http://localhost:8080`
- Backend default dev profile uses H2 in-memory database and enables the H2 console
- Uploaded photos are stored in `uploads/`
- Generated thumbnails are stored in `uploads/thumbnails/`
- Uploaded profile photos are stored in `uploads/profiles/`
- JFIF/JFI uploads are normalized to JPG when possible during storage

## Troubleshooting

If login/register fails:

1. Ensure backend is running on port `8080`.
2. Ensure frontend is running on port `5173`.
3. Restart both servers if needed.
4. Check browser devtools network tab for `/api/auth/*` errors.

If share, like, follow, or comment actions fail:

1. Ensure you are logged in for protected actions.
2. Confirm frontend is using the correct backend URL via `VITE_API_BASE` or Vite proxy.
3. Check that backend CORS allows your frontend origin through `APP_CORS_ALLOWED_ORIGIN_PATTERNS`.

If downloads open in a new tab instead of saving directly:

- Some third-party or cross-origin images may block direct blob downloads in the browser. The app falls back to opening the image URL.

If photos do not persist after restart:

- You are likely on H2 in-memory DB; this is expected.

## Make It Live On Free Services

You can deploy PhotoTribe fully on free-tier tools. The most common setup is:

- Frontend: Vercel or Netlify (free)
- Backend: Render Web Service (free)
- Database: H2 (quick demo) or Neon Postgres (free, persistent)
- Image storage: local filesystem for demo, Cloudinary free plan for production-like usage

Important for production routing:

- Frontend supports `VITE_API_BASE` for deployed backend URLs.
- Example file: `frontend/.env.example`
- Backend CORS is controlled by `APP_CORS_ALLOWED_ORIGIN_PATTERNS`.

### Option 1: Quick Demo Deployment (Fastest)

This is easiest to get online quickly.

1. Deploy backend to Render:
   - Create a new Web Service from your GitHub repo.
   - Root directory: `backend`
   - Build command: `mvn clean package -DskipTests`
   - Start command: `mvn spring-boot:run`
   - Add environment variables for JWT secret and any app configs.
   - You can also use the included `render.yaml` blueprint from repo root.

2. Deploy frontend to Vercel:
   - Import your repo.
   - Root directory: `frontend`
   - Build command: `npm run build`
   - Output directory: `dist`
   - Add env var for backend URL (for example `VITE_API_BASE=https://your-backend.onrender.com/api`).

3. Update backend CORS settings:
   - Allow your deployed frontend domain (Vercel or Netlify URL), not only localhost.

4. Redeploy backend and frontend.

For Netlify instead of Vercel:

- Use `frontend/netlify.toml`
- Set frontend environment variable `VITE_API_BASE`

Important note:

- If backend runs with H2 in-memory DB, data resets on restart/sleep. Good for demo, not persistent.

### Option 2: Free Persistent Deployment (Recommended)

Use this setup for better real-world behavior.

1. Backend on Render (same as above).
2. Frontend on Vercel or Netlify (same as above).
3. Database on Neon (free Postgres):
   - Create Neon project and copy connection URL.
   - Add DB env vars in Render.
   - Configure Spring datasource for Postgres profile.
4. Optional image hosting on Cloudinary:
   - Store uploaded images remotely instead of local server disk.
   - This avoids file loss on container restarts.

## Copy-Paste Deployment Setup

Use this section as a quick setup template.

### 1. Render Backend Settings

Create a new Web Service in Render with these values:

- Runtime: Java
- Region: nearest to your users
- Branch: main
- Root Directory: backend

## Render + Netlify Deployment (Recommended)

Use this exact flow to deploy backend on Render and frontend on Netlify.

### 1. Push latest code

```bash
git add .
git commit -m "Prepare Render and Netlify deployment"
git push origin main
```

### 2. Deploy backend on Render

1. Open Render and choose **New +** -> **Blueprint**.
2. Connect this GitHub repository.
3. Render will detect `render.yaml` and create `phototribe-backend`.
4. In Render service environment variables, set:
   - `SPRING_PROFILES_ACTIVE=dev`
   - `JWT_SECRET_BASE64=<your base64 secret>`
   - `JWT_EXPIRATION_MS=86400000`
   - `APP_CORS_ALLOWED_ORIGIN_PATTERNS=https://<your-site>.netlify.app`
5. Deploy and wait for service status **Live**.
6. Copy backend URL, for example: `https://phototribe-backend.onrender.com`.

Generate a base64 secret locally (PowerShell):

```powershell
[Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes("replace-this-with-a-very-long-random-secret-at-least-32-chars"))
```

### 3. Deploy frontend on Netlify

1. Open Netlify and choose **Add new site** -> **Import an existing project**.
2. Select this repository.
3. Set base directory to `frontend`.
4. Build command: `npm run build`.
5. Publish directory: `dist`.
6. Add environment variable:
   - `VITE_API_BASE=https://phototribe-backend.onrender.com/api`
7. Deploy site.

### 4. Final CORS update on Render

After Netlify gives your final URL, update Render env var:

- `APP_CORS_ALLOWED_ORIGIN_PATTERNS=https://<your-final-site>.netlify.app`

Then redeploy backend once.

### 5. Verify deployment

1. Open Netlify URL.
2. Register a new user.
3. Login and upload a photo.
4. Open admin dashboard and verify moderation/analytics pages load.

### Notes

- `dev` profile uses H2 in-memory database. Data resets on backend restart/sleep.
- For persistent data later, switch to managed database and set datasource env vars.

Do not keep only localhost entries for production.

### 5. Post-Deployment Smoke Test

After both services are live:

1. Open frontend deployed URL.
2. Register a new account.
3. Login with that account.
4. Upload one photo.
5. Confirm photo appears in gallery and remains after refresh.
